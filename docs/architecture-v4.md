# MESA v4 full-cognitive architecture

This document is the canonical architecture for the unreleased v4 API.
`ARCHITECTURE.md` and v3 ADRs remain historical compatibility records.

## Runtime boundary

V4 has one supported storage-writing topology: one `combined` process owns
SQLite, LanceDB and Kùzu. API admission and background consumers run in that
process. A second writer must never mount the same storage root.

V3 remains available as the lexical-core compatibility surface and keeps its
existing API/worker topology. V3 and v4 must not write the same physical
storage directories concurrently.

## Security and catalog hierarchy

The v4 security boundary is a tenant, not an agent:

```text
Principal
  └── Tenant
      └── Workspace
          └── Dataset
              └── Document
                  └── immutable DocumentRevision
                      └── SourceChunk
```

Roles are `OWNER`, `WRITER` and `READER` at tenant, workspace or dataset
scope. Roles inherit downward. Purge and rollback require explicit dataset
permissions. Agent IDs represent compute/persona context and do not grant
tenant access.

An authenticated session is generated by the server and binds one principal,
tenant, workspace, agent and immutable set of datasets. Insert, search,
context, status, replay and rollback derive their scope from this record.

API credentials use `key_id.secret`. Only the salted scrypt digest is stored;
plaintext is returned once by `mesa-v4-admin issue-key`. Multiple keys may
belong to one or more principals and can be revoked or rotated independently.

## Canonical ingestion

`POST /v4/memory/insert` durably admits exact source content and creates:

- one `MemoryCandidate`;
- one `memory_mutations` ledger row;
- one `pipeline_runs` row plus append-only events;
- ordered SQL, vector and graph outbox lanes.

The candidate explicitly carries tenant, workspace, dataset, document,
revision, chunk, source reference, evidence span, agent/session, pipeline,
extraction and embedding versions. Authorization fields are never read from
free-form metadata.

Tier-3 validation is a mandatory gate. A rejected mutation may keep its raw
audit source, but it cannot create an active SQL, vector, entity, edge or
assertion artifact.

## Pipeline and failure model

The legal success path is:

```text
QUEUED → RUNNING → EXTRACTED → VALIDATED → PROJECTING → COMMITTED
```

Failure/recovery states are `RETRY_PENDING`, `DLQ`, `ROLLING_BACK`,
`ROLLED_BACK` and `BLOCKED`. State changes use compare-and-swap and append an
event. Claims use a random fencing token, bounded lease, heartbeat and bounded
retry. A worker that loses its token cannot finalize the claim.

Projection prerequisites are enforced in the database:

```text
SQL → vector → graph
```

The projector only claims a later lane when its predecessor is complete.
Every lane is idempotent by mutation and physical artifact identity.

## Canonical entity and Graph V2

Entity IDs use a fixed UUID5 namespace:

```text
UUID5(namespace, tenant_id + entity_type + identity_key)
```

`identity_key` is the normalized ontology URI when supplied; otherwise it is
the Unicode NFKC, whitespace-normalized and case-folded canonical name.
Resolution order is ontology/external ID, registered alias, canonical name,
then create. Collisions produce a deterministic redirect; historical rows are
retained.

SQLite is authoritative for `v4_entities` and immutable `v4_assertions`.
Kùzu is an idempotent projection. Assertions include subject, predicate,
entity or literal object, document/revision/chunk source, evidence span,
jurisdiction, authority, confidence, validity interval, mutation and pipeline
run. Assertion identity includes predicate, source and temporal context, so
different relations between the same entities remain distinct.

`CONTRADICTS` and `SUPERSEDES` connect assertions and preserve provenance.
PageRank only publishes observation telemetry; it cannot quarantine,
tombstone, delete or otherwise change retrieval eligibility.

## Artifact ownership and rollback

`artifact_registry` represents physical SQL, vector and graph artifacts.
`artifact_sources` allows several mutations/source chunks to own the same
entity or vector.

Rollback removes only the requesting run's source ownership. A shared artifact
remains active while another source owns it. The last-owner path writes the
SQLite retrieval tombstone first and schedules idempotent vector/graph cleanup.
An unfinished cleanup leaves the run `BLOCKED`, never falsely committed.

A new document revision does not erase history. Prior assertions become
`SUPERSEDED`. A legal purge retracts the requested source and removes only
artifacts that have no remaining active owner.

## Reconciliation and health

Reconciliation is bidirectional:

- ledger artifact missing from a store: requeue the missing lane;
- physical artifact without registry/ownership: quarantine it in the registry
  and enqueue cleanup.

`/health` exposes projection backlog, dead letter, stuck leases, cleanup
backlog, blocked cleanup, ownerless registry rows and shared-owner counts.
`/metrics` exports the same content-free aggregate gauges.

## Retrieval

All SQL, vector and graph lanes receive the authorized dataset filter before
ranking. V4 combines rank positions using real reciprocal rank fusion. A
bounded legal rerank may then use jurisdiction, validity time, authority and
assertion confidence. Entity ID is the deterministic final tie-break.

Responses include document, revision, chunk, evidence, assertion, pipeline and
embedding provenance. PageRank is not a retrieval eligibility signal.

## Migration and release

V3 storage is never converted in place. The supported sequence is:

1. stop writers and create a manifest/checksummed backup;
2. restore or copy into a new empty staging root;
3. rebuild v4 SQLite/vector/graph artifacts from raw sources;
4. run parity and authorization checks;
5. atomically cut over to the staged root;
6. retain the previous root for explicit rollback.

Production remains `NO-GO` until migration/restore, v4 Compose, crash,
saturation, cross-tenant concurrency, benchmark and 24-hour soak gates all
pass with production-like storage and the selected model provider.
